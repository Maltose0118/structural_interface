# Structural Interface (SI)

## Design Specification

---

# 1. Overview

Structural Interface (**SI**) is a unified interface mechanism built on **Structural Member Matching** and **C++26 static reflection**.

A single interface definition gives rise to two views:

* **Compile-time View**

  * `si::satisfies<T, I>`
  * Serves as generic constraints
  * Zero runtime overhead

* **Runtime View**

  * `si::existential<I>`
  * `si::existential_move_only<I>`
  * `si::existential_ref<I>`
  * Type-erased dynamic polymorphism

The interface is defined once; all representations are automatically generated by the compiler through reflection.

The entire system relies on **no macros, no external code generators** — only C++26 static reflection.

---

# 2. Design Philosophy

SI describes **syntactic interfaces** rather than semantic interfaces.

The interface specifies only:

> Which members a type must publicly expose.

It does **not** prescribe:

> What behavior or semantics those members should have.

For example:

```cpp
struct Drawable {
    void draw() const;
};
```

means a type must expose:

```cpp
void draw() const;
```

Whether:

* `draw` actually draws anything;
* it modifies object state;
* it is thread-safe;
* it satisfies any other semantic constraints;

is not SI's concern.

This follows the same terminology as C++ Concepts:

* **satisfies**: meets syntactic requirements;
* **models**: meets both syntactic and semantic requirements.

SI only checks **satisfies**.

---

# 3. Interface Definition

An interface is defined as an ordinary `struct`.

```cpp
struct Drawable {
    void draw() const;
};

struct Geom {
    int width;
    int height;
};
```

No base class is required.

No macros are required.

Future support may include:

```cpp
[[si::interface]]
```

as an optional annotation.

---

# 4. Allowed Members

Interfaces may contain:

* Non-static member function declarations (no function body);
* Non-static data members;
* Nested interfaces;
* Interface composition (inheriting other interfaces).

Interfaces must **not** contain:

* Constructors;
* Destructors;
* Static members;
* `friend` declarations;
* `using` declarations;
* Type aliases;
* Function definitions (bodies);
* Virtual base classes.

Interface members must be public.

Matching members in concrete types must also be accessible in the current context. By default, private / protected members do not participate in matching and cannot have their access bypassed by runtime thunks.

---

# 5. Structural Satisfaction

The system automatically generates:

```cpp
namespace si {

template<class T, class I>
concept satisfies = /* compiler generated */;

}
```

The concept holds if and only if **every member of the interface** has a corresponding member in type `T`:

```cpp
si::satisfies<T, I>
```

---

## 5.1 Function Matching

Member functions are matched based on:

* Name;
* Parameter types;
* Return type;
* cv-qualifiers;
* Ref-qualifiers (`&` / `&&`);
* `noexcept`.

All of the above must match exactly.

Default arguments do not participate in matching.

If the interface contains multiple overloads:

```cpp
struct Drawable {
    void draw();
    void draw(int);
};
```

Each overload is matched independently.

---

## 5.2 Data Member Matching

Data members are matched based on:

* Same name;
* Same type.

For example:

```cpp
struct Geom {
    int width;
};
```

can be satisfied by:

```cpp
struct Window {
    int width;
};
```

Data members serve purely as structural requirements.

Their semantics are not SI's concern.

---

## 5.3 Function Templates

Function templates participate in matching only if they can be instantiated to produce an exactly matching member function.

For example:

```cpp
template<class T>
void draw(T);
```

can satisfy:

```cpp
void draw(int);
```

but cannot satisfy:

```cpp
void draw();
```

---

# 6. Interface Composition

Interfaces can be composed by inheriting other interfaces:

```cpp
struct Shape
    : Drawable,
      Transformable
{
};
```

This means:

```text
Shape = Drawable ∩ Transformable
```

i.e., both parent interfaces must be satisfied simultaneously.

A composed interface is still an ordinary interface definition.

It incurs no runtime inheritance relationship.

---

## 6.1 Conflict Detection

If multiple parent interfaces contain:

* Data members with the same name but different types;
* Member functions with the same name but different function types;

the composition is **illegal**.

The compiler produces a diagnostic.

---

# 7. Runtime Existential Types

For each interface `I`, the system automatically generates:

```cpp
si::existential<I>

si::existential_move_only<I>

si::existential_ref<I>
```

These differ only in ownership semantics.

| Type | Ownership | Copy | Move |
| ---- | --------- | ---- | ---- |
| `existential` | Owns the object | Calls object copy constructor | ✔ |
| `existential_move_only` | Owns the object | ✘ | ✔ |
| `existential_ref` | Non-owning reference | Copies the reference | ✔ |

---

# 8. Runtime Representation

SI uses a unified Runtime Interface Metadata scheme.

For every concrete type `T` satisfying:

```cpp
si::satisfies<T, I>
```

the system automatically generates unique static metadata:

```cpp
template<class I>
struct interface_metadata {

    // Lifetime management

    void (*destroy)(void* object);

    void (*copy_construct)(
        void* dst_storage,
        void** dst_object,
        const void* src_object
    );

    void (*move_construct)(
        void* dst_storage,
        void** dst_object,
        void* src_object
    );

    // Interface member function thunks
    // ...

    // Field offsets
    // ...
};
```

Each `(Interface, Concrete Type)` pair corresponds to exactly one static `constexpr` metadata instance.

It exists exactly once across the program lifetime.

`interface_metadata` is not merely a function table. It describes the complete runtime information of how a specific concrete type implements a given interface, including lifetime management, method dispatch, field access, and SBO / heap object management strategy.

---

# 9. Object Representation

## 9.1 Owning Existential

Owning existential types use Small Buffer Optimization (SBO):

```cpp
template<class I>
class existential {
    /* generated interface member proxies */

    _si_existential_details<I> _si_details_;
};

template<class I>
struct _si_existential_details {
    alignas(ImplementationDefinedAlignment)
    std::byte storage[ImplementationDefinedStorage];

    void* object;

    const interface_metadata<I>* metadata;
};
```

Runtime layout:

```text
+----------------------------------+
| inline storage (SBO)             |
| object pointer                   |
| interface_metadata pointer       |
+----------------------------------+
```

Small objects:

* `object == storage`
* The object is placement-`new`-ed into `storage`

Large objects:

* `object = new T(...)`
* `storage` is unused

Thus method thunks always need only:

```cpp
static_cast<T*>(object)
```

without needing to check whether the object uses SBO.

## 9.2 Reference Existential

The reference existential stores the object address:

```cpp
template<class I>
class existential_ref {
    /* generated interface member proxies */

    _si_ref_details<I> _si_details_;
};

template<class I>
struct _si_ref_details {

    void* object;

    const interface_metadata<I>* metadata;
};
```

It does not own the object's lifetime.

---

# 10. Construction

Example:

```cpp
Circle c;

si::existential<Drawable> e = c;
```

For a small object, this is equivalent to:

```cpp
new(storage) Circle(c);

object = storage;
metadata = &metadata_for<Drawable, Circle>;
```

For a large object:

```cpp
object = new Circle(c);

metadata = &metadata_for<Drawable, Circle>;
```

Therefore:

* Lvalues invoke the object's copy constructor;
* Rvalues invoke the object's move constructor;
* Small objects use local storage;
* Large objects use heap allocation.

---

# 11. Copy / Move

Copy:

```cpp
existential a = Circle{};
existential b = a;
```

Performs:

```cpp
a.metadata->copy_construct(
    b.storage,
    &b.object,
    a.object
);

b.metadata = a.metadata;
```

Move:

```cpp
existential b = std::move(a);
```

Performs:

```cpp
a.metadata->move_construct(
    b.storage,
    &b.object,
    a.object
);

b.metadata = a.metadata;
```

Ultimately the underlying object's copy or move constructor is called.

The small-object version may be generated as:

```cpp
template<class T>
void copy_construct_small(
    void* dst_storage,
    void** dst_object,
    const void* src_object
) {
    new(dst_storage) T(*static_cast<const T*>(src_object));
    *dst_object = dst_storage;
}
```

The large-object version:

```cpp
template<class T>
void copy_construct_large(
    void*,
    void** dst_object,
    const void* src_object
) {
    *dst_object = new T(*static_cast<const T*>(src_object));
}
```

The existential itself does not need to know whether the object is small or large.

The difference between small and large objects is hidden in the lifetime management functions pointed to by `interface_metadata`.

Thus:

* `existential<I>` requires the underlying object to satisfy `std::copy_constructible`.
* `existential_move_only<I>` requires the underlying object to satisfy `std::move_constructible`.

---

# 12. Destruction

Destruction:

```cpp
~existential()
```

Performs:

```cpp
metadata->destroy(object);
```

The small-object destroy thunk calls the destructor.

The large-object destroy thunk calls the destructor and frees heap memory.

The existential itself does not need to know whether the object is on the heap.

---

# 13. Method Dispatch

Value-semantic calls to interface member functions are realized at two layers:

* `existential<I>` / `existential_ref<I>` injects member variables with the same names as the interface members;
* The types of these member variables are compiler-generated callable proxies, which dispatch to the real object through thunks in `interface_metadata`.

Specifically, for:

```cpp
struct Drawable {
    void draw() const;
};
```

the system generates a corresponding slot in `interface_metadata<Drawable>`:

```cpp
template<>
struct interface_metadata<Drawable> {
    void (*destroy)(void* object);
    void (*copy_construct)(void* dst_storage, void** dst_object, const void* src_object);
    void (*move_construct)(void* dst_storage, void** dst_object, void* src_object);

    void (*draw)(const void* object);
};
```

And a thunk is generated for each concrete type satisfying `Drawable`:

```cpp
template<class T>
void draw_thunk(const void* object) {
    static_cast<const T*>(object)->draw();
}
```

The system injects into `si::existential<Drawable>` a member variable equivalent to:

```cpp
si_member_function_proxy<
    existential<Drawable>,
    &interface_metadata<Drawable>::draw
> draw;
```

Thus user code can write directly:

```cpp
si::existential<Drawable> obj = Circle{};

obj.draw();
```

Here `obj.draw()` is not a member function call, but a member object call:

```cpp
obj.draw.operator()();
```

That is, C++26 static reflection only needs to inject member variables; the member variable's type provides `operator()`, preserving the value-semantic `obj.draw()` syntax.

Interface member functions are dispatched through thunks in `interface_metadata`:

```text
callable member variable
        │
        ▼
owner existential object
        │
        ▼
interface_metadata
        │
        ▼
generated thunk
        │
        ▼
T::member(...)
```

Each call incurs exactly one function pointer indirection.

Performance is on par with traditional virtual function calls.

---

## 13.1 Injected Callable Member

To make `obj.draw()` work, the system generates a callable member variable type for each interface member function.

Conceptually:

```cpp
template<class Owner, auto Slot>
struct si_member_function_proxy {
    void** object;
    /* thunk slot */

    decltype(auto) operator()() const {
        return /* thunk */(*object);
    }
};
```

The compiler then injects this proxy as a member variable into the existential family:

```cpp
template<class I>
class existential {
    _si_existential_details<I> _si_details_;

public:
    si_member_function_proxy<
        existential<I>,
        /* slot for I::draw */
    > draw;
};
```

The proxy is bound by library implementation to the object pointer slot and the corresponding thunk in `_si_details_`.

The implementation can adopt any of the following strategies:

* The proxy member holds a pointer to the `_si_details_` object slot;
* The proxy member holds a pointer to the owner;
* The proxy member recovers the owner from its own address via a compiler-known member offset;
* The compiler treats the proxy as a special injected member, allowing access to the enclosing object.

The first strategy is the most straightforward, but adds one pointer of storage per interface member.

The latter two strategies can achieve zero additional storage — potentially using `[[no_unique_address]]` — but depend on implementation support for injected members.

In the C++26 static reflection model, the generator enumerates the member declarations of interface `I` and injects corresponding callable member variables into `existential<I>`, `existential_move_only<I>`, and `existential_ref<I>`. Reflection handles "reading the interface shape"; member variable injection handles "giving the wrapper callable entry points with the same names."

The key requirements are:

* For each member function in the interface, generate a callable member variable with the same name;
* Preserve parameter types, return types, cv/ref qualifiers, and `noexcept`;
* Support generation of overloadable callable proxies when the interface contains overloads;
* `const` members are dispatched through `const void*` thunks;
* `&`-qualified members are dispatched on lvalue objects;
* `&&`-qualified members are dispatched via `std::move(*static_cast<T*>(object))`;
* The existential's public API should look and feel like a value-semantic object of interface `I`.

For example:

```cpp
struct Consumable {
    void consume() &&;
};
```

The generated `existential<Consumable>` supports:

```cpp
si::existential<Consumable> value = Token{};

std::move(value).consume();
```

The corresponding thunk performs on the real object:

```cpp
std::move(*static_cast<T*>(object)).consume();
```

This ensures that SI's runtime view is not merely "a box that can be explicitly dispatched," but a value-semantic object with member entry points that mirror the interface.

---

# 14. Field Access

Interface data members are also realized through member variable injection, but the runtime wrapper exposes **raw field pointers**.

```cpp
obj.width
```

In the current library implementation, data members are injected into a generated base class of the existential itself. `obj.width` has a raw pointer type, e.g. `int*` or `std::string*`.

Therefore users explicitly dereference to access fields:

```cpp
*(obj.width) = 10;
int value = *(obj.width);
```

Field pointers are bound to the named member of concrete type `T` when the wrapper is constructed. After copy / move, they are re-bound to the object fields held by the new wrapper.

The access path:

```text
obj.width
        │
        ▼
T::field
```

Is equivalent to:

```cpp
constexpr auto member = /* reflected T::width */;
obj.width = std::addressof(static_cast<T*>(object)->*member);
```

This design keeps expression semantics unambiguous: the field name corresponds to a field pointer, and the library does not simulate reference members.

---

# 15. Storage Requirements

Owning existential types use an implementation-defined SBO local storage.

Implementation-defined:

* `ImplementationDefinedStorage`
* `ImplementationDefinedAlignment`

If:

```cpp
sizeof(T) <= ImplementationDefinedStorage

alignof(T) <= ImplementationDefinedAlignment
```

then the object is stored in local SBO storage.

Otherwise it is stored on the heap.

SI automatically selects the appropriate lifetime management functions for small or large objects.

---

# 16. Reflection-Based Generation

The system relies entirely on C++26 static reflection.

The compiler automatically generates:

* `si::satisfies`
* `interface_metadata`
* Callable member variable proxies
* Field pointer member variables
* Lifetime management functions
* Member function thunks
* Field offsets
* `existential`
* `existential_move_only`
* `existential_ref`
* Diagnostic information

No macros.

No external code generators.

---

# 17. Diagnostics

When structural matching fails:

```text
error:
'Circle' does not satisfy
'si::satisfies<Drawable>'

missing member:

void draw() const
```

Interface composition conflict:

```text
error:

interface 'Shape'

contains conflicting member:

width
```

Diagnostics should directly indicate the unsatisfied member.

# 18. Performance Guarantees

| Operation | Overhead |
| --------- | -------- |
| `si::satisfies` | Compile-time |
| Static dispatch | Zero runtime overhead |
| Existential method call | One function pointer indirection |
| Existential field access | One fixed offset |
| Existential small-object construction | Constructor call |
| Existential large-object construction | One heap allocation + constructor call |
| Existential copy | Object copy constructor call |
| Existential move | Object move constructor call |
| Existential destruction | Object destructor call, heap deallocation if needed |

Small-object path incurs **no heap allocation**.

The large-object path follows the same SBO strategy as modern C++ type-erasure facilities such as `std::function` and `std::any`.

---

# 19. Non-goals

SI does not attempt to:

* Describe interface semantics;
* Validate `models`;
* Replace class inheritance hierarchies;
* Implement virtual inheritance;
* Rely on RTTI;
* Provide a complete reflection framework;
* Automatically derive interfaces.

---

# 20. Example

```cpp
#include <structural_interface.hpp>

#include <vector>

struct Drawable {
    void draw() const;
};

struct Circle {
    void draw() const {
        // ...
    }
};

struct Rectangle {
    void draw() const {
        // ...
    }
};

void render(si::satisfies<Drawable> auto&& obj) {
    obj.draw();
}

int main() {

    render(Circle{});

    std::vector<si::existential<Drawable>> objects;

    objects.emplace_back(Circle{});
    objects.emplace_back(Rectangle{});

    for (auto& object : objects)
        render(object);
}
```

In this example:

* `render(Circle{})` uses static dispatch;
* `render(object)` uses existential dynamic dispatch;
* Both share the same interface definition, with no need for a separately declared Concept, abstract base class, or type-erasure wrapper.

---

# 21. Architecture Summary

The overall architecture of SI is:

```text
                Interface Definition
                        │
            Reflection generates
                        │
        ┌───────────────┴───────────────┐
        │                               │
   satisfies<T,I>              Runtime View
        │                               │
 Static Dispatch        ┌──────────────┴──────────────┐
                        │                             │
              interface_metadata         injected member proxies
                        │                             │
                        └──────────────┬──────────────┘
                                       │
                              existential family
```

The runtime object is uniformly represented as:

```text
+----------------------------------+
| inline storage (SBO)             |
| object pointer                   |
| interface_metadata pointer       |
+----------------------------------+
```

Where:

* Small object: `object` points to `storage`
* Large object: `object` points to a heap object

`interface_metadata` is the unique static `constexpr` metadata for each `(Interface, ConcreteType)` pair, responsible for lifetime management, method dispatch, and field access.

This model balances performance, generality, and implementation complexity, and is consistent with modern C++ type-erasure design patterns.
